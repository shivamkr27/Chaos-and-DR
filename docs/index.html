What this is

A resilience platform across two AWS regions that deliberately breaks itself and proves it recovers automatically — pod kills, network degradation, CPU stress, and a full region outage. Everything here runs on real infrastructure at effectively $0/month: a t3.micro K3s node per region instead of managed Kubernetes, a Cloudflare Worker instead of paid DNS failover, and chaos triggered by hand instead of a full chaos-engineering platform (that part OOM'd on a 1GB node — the honest failure that led to the simpler, working version you see here).

Frequently asked
Why a Cloudflare Worker instead of Route 53?

Route 53 failover costs money per health check and has a 60-90s minimum TTL. The Worker is free, checks health on every request with zero propagation delay, and doesn't need an owned domain. For a real production system with a domain already in place, Route 53 is the more conventional answer.

Why K3s instead of EKS?

K3s exposes the full Kubernetes API — same manifests, same kubectl — at zero cost on a t3.micro. EKS charges roughly $72/month just for the control plane. The concepts transfer directly; swapping to EKS is a kubeconfig and IAM change, not a redesign.

What happens to the database during a region failover?

Primary runs RDS PostgreSQL. DR runs the app without a database — it stays healthy and serves traffic, but write operations degrade gracefully. Cross-region RDS replication needs backup retention that costs money outside the free tier, so it's intentionally left off for this demo (the Terraform module for it exists, just disabled).

What would change at real scale?

Active-active with conflict resolution at the data layer, Velero for cluster-state backup, a real chaos-engineering platform (Gremlin, chaos-mesh) instead of scripted kubectl/tc commands, and a written hypothesis + blast-radius plan before every experiment.
